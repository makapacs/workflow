About

Overall

WorkFlow is a tool used to build assets, deploy and configure these assets on drupal instances among other things

Each asset contains any number of artifacts with one of the two types - widget and theme, but is not limited to only those two types. You can add any type at any given type

Each asset has actions that are tasks that can be applied at the time of installation and configuration of asset

Each asset has a blueprint or also known as make file. The blueprint specifies all the artifacts to include in the asset, the associated actions among other things for whenever you build, deploay or configure asset. There are 3 main ways to supply the blueprint from command line:

  1. --asset: by providing a name of asset, WorkFlow will locate the blueprint following conventions(see 'conventions' section) and environment configurations (see 'External Environment Configurations' section)
  2. --package: by providing dir of a package or compressed tar/zip file, it will take the make file located in the package directory
  3. --make: by specifying directly the make file is another way
You only need to do one of the above to provide blueprint for tool to execute

Deployment Modes

When deploying asset, you have a choice to deploy it in one of the 3 forms - dev, test or stage. Each of the forms differ in how assets are downloaded and setup into your Drupal instance. In the following, is quick overview:

  • DEV: artifacts are linked to from the working directory into the drupal instance for quick easy changes
  • TEST: artifacts are copied from package provided. This is perfect for testing before shipping asset
  • STAGE: artifacts are cloned from github.

Installation

How To Install?

In the following are the steps to install Artifact WorkFlow Drupal tool

  1. Download
    Clone the tool from github repo - https://github.com/makapacs/workflow
  2. Incorporate With Drush
    Update $options['include'] to include path to the tool with '/commands' in Drush configuration file drushrc.php (usually in $HOME/.drush dir) $options['include'] = array('/path/to/Artifact Workflow tool/commands');

  3. choose one of the two:

    • Grab WorkFlow Sandbox from https://github.com/makapacs/workflow-sandbox and start develop. NO CONFIGURATION NEEDED
    • Configure Your Environment
      Specify your Environment(working dir, distribution dir, etc.) in the workflow.drushrc.php and place it in to DRUSH_ROOT dir as described in 'Configuration' section
  4. add additional configurations such db user, db password,etc to alias. Please, see Alias Configuration section

Terminology

Terminology Terms

  • Asset
    Asset is the product that combines together different artifacts and actions to provide some functionality, solve some problem. It is the deliverable you ship. Some examples of assets are theme, slider, blog, newsletter etc. or combined multiple of the items solving different problems into one asset deliverable and deployable on another Drupal instance. It's typical to have one main artifact and rest of them are dependent artifacts in one asset. Based on this main artifact type, we assign the artifact type to an asset be more descriptive
  • Artifact
    There are two types of artifacts - theme, widget that are all combined together to form an asset to provide one or many functionality such as slider, blog, calendar, etc.
  • Theme
    Theme is an artifact located at sites/all/themes directory by default. There is usually a dependent a custom module another artifact to provide functionality for a theme. There may else be base theme another artifact. All of these combined create asset of type theme
  • Widget
    Widget is an artifact that contains one or more modules to provide some functionality such as calendar, blog, slider, etc. By default, all widgets are stored in sites/all/modules dir It can also be referred as plug-in or add-on
  • Action
    Actions can be tasks such us configure video, install dependencies, etc. that are applied to Drupal instance during deployment/configuration stage. The tasks can also be individual actions such as check the uncommitted changes in working directory, scrap email addresses from file,etc. Actions are also any Drush command. The purpose of actions is to automate repetitive tasks that can be reused by declaring in blueprint(make file) and executing them all at once or per requirements. Actions can also be shared among assets and imported into deliverable package at the build process. See "Actions" section for details
  • Package Deliverable of asset that can be compressed in tar/zip format. It contains all the artifacts all arranged per directories - modules, themes

The 'prefix' is configured in the Tool configurations. See section "Configurations"

Depending if the flags '--min', '--ver', '--branch' is provided, the package name is going to be constructed in the order from more specific to general as demostrated above

Blueprint

About Blueprint

every artifact such as widget or theme has its own make file defining all dependencies, configuration that is used to build, deploy,etc

Where is Make file located?

The make files is located in dir specified by environment configuration 'make_root' plus '/' plus 'type' of the artifact. The name of the make file is the same as the widget package or less specific

Alias - Site Configurations

Alias Extended

The alias configurations are extended with the following configuration options:

  • db-user: The database username for the current drupal installation
  • db-pwd The database user password for the current drupal installation
  • db-name The database name for current drupal installation
  • db-url the database url for current drupal installation. If specified, than it will be used instead of db-user,db-pwd,db-name (i.e. $db_url = 'mysql://user-name:pass@127.0.0.1/db_name)
  • make-file full path to the make file used for initial installation of Drupal if one is not provided at the time of execution 'site-install' command

Important to note, that alias configurations - db-name, db-user, db-pwd will be overritten with ones user specifies at the time of execution if any

Artifacts

About Artifacts

The artifacts is what this is about. The WorkFlow tool builds packages from artifacts, deploys them, configures artifacts on instances and more. By default, there are two artifacts - widget and theme, but is not limited to only two. You can declare others(see Configurations section)

In the following is artifact attributes:

  • name: The name of the artifact. It has to be unique and the same across all declarations(aliases can be configured)
  • type The type of the artifact such as widget, theme, etc. All artifacts without type declared is of type 'widget'
  • repo The url of github account containing the repository of an artifact. Make sure there is forward slash at the end(ie. https://github.com/git-account/)
  • repo_name the repository name in GitHub storing an artifact. The 'repo'/'repo_nam' combains full path to clone artifact
  • branch for developers its the branch in the GitHub repo. For business people, it may be the package of the deliverable(i.e. BASIC, PLUS, ULTIMATE)
  • ver The tag name of the git repository. This is used to version each artifact.
  • final_name the final name of artifacts as it is stored in the 'site/all/modules','sites/all/themes' directories or in devliveralbe package
  • work_dir the path to local working dir of the artifacts. It overrides the general 'work_dir' specified in environment configuration
  • work_dir_name the dir name of artifact in the local working directory.

Important to note, that one artifact will contain multiple dependencies that are other artifacts(with their own blueprints - make files). They all need to be declared the same way in the blueprint - make file of that main artifact being build, deployed or configured

Declaring In Blueprint

Each artifact can be declared in the make file as following:

widget[commerce][name]=commerce
widget[commerce][order]=1
widget[commerce][final_name]=designssquare_com_widget_commerce
widget[commerce][repo]=https://github.com/kapasoft-drupal-modules/
widget[commerce][repo_name]=commerce
widget[commerce][ver]=v.0.1
widget[commerce][work_dir]=/path/to/local/work/dir
widget[commerce][work_dir_name]=commerce_widget
... theme[bazar][name]=bazar
theme[bazar][order]=0
theme[bazar][final_name]=bazar
theme[bazar][repo]=https://github.com/kapasoft-drupal-modules/
theme[bazar][repo_name]=bazar-theme
theme[bazar][ver]=v.0.3.1
theme[bazar][work_dir_name]=bazar
...
YOUR-TYPE[index][name]=name
YOUR-TYPE[index][order]=0
...

Here, we have declared two different type of artifacts. One is widget - commerce and another is theme - Bazar. When declaring artifacts in the make file there additional attribute 'order' that provides order of processing the artifacts, because some may depend on another.

Important to note. The index of the artifact has to be the name

You may notice that not all attributes are declared. Some attributes like 'work_dir' is declared in the environment configurations. Unless its different for specific artifact, you don't have to re-declare it in make file. All other attributes of artifact can be declared in the external artifact configuration once and only referred in the make file

Declaring In External Configurations

After a while, you will get tired re-declaring the artifacts in the make file, so to avoid this redundant task, there is available artifact external configurations to declare each artifact once and then reference in make file by name instead. In the make file you can also override any of the attribute such as 'ver',etc, since any attributes specified in make file will mask out ones declared in the external artifact configurations

To declare artifacts externally, create a file 'artifacts.drushrc.php' and place it any path Drush searches for. Here is the sample format of 'artifacts.drushrc.php':

//declaring artifacts
$artifacts = array();
$artifacts['lib'] = array(
'name' => 'lib',
'repo' => /path/to/repo,
'repo_name' => 'designssquare-utils',
'final_name' => 'designssquare_com_lib',
'work_dir_name' => 'designssquare_lib',
);
$artifacts['sample_data'] = array(
...
);

Environment Configurations

Location of Configurations

The configurations are externalized into workflow.drush.inc in DRUSH_ROOT dir($HOME/.drush/ dir) or any location DRUSH searches

Available Environment Configurations

  1. work_dir
    full path of working directory that contains all of your widgets such as themes, modules, sites, etc categorized into dir 'themes', 'modules','sites',etc. This is where you do the the development. The packaging is done from the working directory. We recommend each of the widgets keep in its own git repository. If not specified, then defaults to directory - $HOME/workflow/work-dir
  2. make_root
    full path to directory that contains the Blueprints - make files for each of the artifacts such as theme, module, site, etc. Just like working dir this directory is categorized into directories per widget type such as themes, modules, sites, etc. So, for example, all make files for modules are in dir 'make_root/modules' and make files for themes are in dir 'make_root/themes' and so on per artifact type. If 'make_root' is not specified in environment configurations, then it defaults to location - $HOME/workflow/blueprints
  3. dist
    full path to directory that contains packages built & deliverable to client ready to be imported in another instance. For each artifact there is a separate dir. The artifact dir name is combined with 'prefix'-'artifact type'-'artifact name'. The artifact directory contains all of the packages versioned that are build so far.
  4. package_prefix
    The package_prefix specifies the prefix added to the package name. For example, all pacakge at designssquare.com is prefixed with 'designssquare-com-'
  5. db_user
    the database user used for site installation, etc if not specified at site alias configurations or at the time of execution
  6. db_pwd
    the database user password used for site installation, etc if not specified at site alias configurations or at the time of execution
  7. make_file
    the default make file for site installations, for shared actions,etc. If not specified in environment external configurations with 'make_file', then it defaults to make_root/build-drupal-core.make

Formatting Environment Configurations

The configurations need to be formatted as array of name '$workflow_config' as following:

$workflow_config = array();
$workflow_config['work_dir'] = '/path/to/work/dir';
$workflow_config['make_root'] = '/path/to/blueprints';
$workflow_config['dist'] = '/path/to/distribution/dir';
$workflow_config['package_prefix'] = 'designssquare-com-';
$workflow_config['make_file'] = 'path/to/default-make-file.make';
$workflow_config['db_user'] = 'username';
$workflow_config['db_pwd'] = 'password';

Available Artifact Configurations

  1. type
    declare the available types of artifacts. By default, it declares - theme, widget types.

Formatting Artifact Configurations

The artifact configuration is an array of name $artifact_types containing all available artifact types

$artifact_types = array();
$artifact_types[] = 'widget';
$artifact_types[] = 'theme';

Conventions

Documentation

its assumed that each widget has its own documentation located at '/doc' dir. It is included in the package and moved into Drupal instance at the deployment

Package Naming

Package naming is combined with 'prefix'-'type'-'name'-'branch'-'min'-'ver'. Each widget also has its own make file and it is named the same as package name or if not present, then it looks up next less specific name in the following order:

  • 'prefix'-'type'-'name'-'branch'-'min'-'ver'
  • 'prefix'-'type'-'name'-'branch'-'min'
  • 'prefix'-'type'-'name'-'branch'
  • 'prefix'-'type'-'name'

So, you have to have at least one general make file 'prefix'-'type'-'name' for each artifact in order to process artifact

Working Dir

In your environment configurations, you would specify the working directory of all artifacts such as themes, widgets, sites or any type of artifact you like to develop. It is a convention that each of type of artifact is in its own directory in the working directory, so widgets will be in 'work_dir/widgets' and themes are in 'work_dir/themes' and so on for any other type. You can overwrite this by specifying different work dir for each artifact in the make file with attribute 'work_dir'. If work_dir is not specified in the environment configurations, then defaults to location - $HOME/workflow/work-dir

Distribution Dir

This directory contains packages build and deliverable to client ready to be imported in another instance. Just like work directory, for each artifact there is a separate dir (i.e. themes, widgets,etc). The specific artifact dir name is combined with 'prefix'-'artifact type'-'artifact name'. The artifact directory contains all of the packages versioned that are build so far. Note, prefix can be configured in external env configurations(please, see configurations section) If 'dist' is not specified in environment configurations, then it defaults to location - $HOME/workflow/dist

Blueprints Dir

This directory contains blueprints for all artifacts. Just like work directory, for each artifact there is a separate dir (i.e. themes, widgets,etc). The specific artifact name is combined with 'prefix'-'artifact type'-'artifact name'. Note, prefix can be configured in external environment configurations(please, see configurations section). In environment configurations, you can specify the blueprint root dir via 'make_root'. If 'make_root' is not specified in environment configurations, then it defaults to location - $HOME/workflow/blueprints

Actions Overview

About Actions

actions are some tasks that was created with intention to reuse. The WorkFlow comes with several already supported actions such as conifg-editor, config-audio,etc. Actions can be any drush command as well. Actions can be declared and reused in the make file The actions declared in the default make file(see external env configurations) are shared for all artifacts

Declaring Actions

The actions are declared in the blueprint(make file) as following

action[load][name]=enable_artifact
action[load][order]=0
action[load][tag]=init
action[load][param][]=designssquare_lib
action[load][param][]=designssquare_lib_sample_data
action[load][param][]=espresso
action[default_theme][name]=set_default_theme
action[default_theme][order]=3
action[default_theme][tag]=init
action[default_theme][param][]=espresso
 Here, action named 'load' enables two modules - designssquare_lib, designssquare_lib_sample_data and theme "espresso". Afterwards, the action named 'default_theme' sets the default theme to be 'espresso'

Solo Action

one way to execute actions is via drush command 'execute-action'[ea]. This executes one action at the time. Another way, to call single action is via drush command 'configure-artifact'[ca] and specify flag '--action=NAME-OF-ACTION' as described in the blueprint(make file)

Multiple Actions

The best way to call multiple actions is using 'tags'. When declaring actions, add tag, so, afterwards, you can call all actions with certain tag as following drush ca ARTIFACT @ALIAS --tag=dev,init This will call all actions with tag 'dev' and 'init'.

In case, you like to call more than one but without tag mechanism, then you can specify list of actions themselves to execute via flag 'action' as following: drush ca ARTIFACT @ALIAS --action=ACTION-NAME1,ACTION-NAME2

Sharing Actions

there may be actions that you like to share among the artifacts. So, you don't have to re-declare those actions for each artifact, you can put them in the default make file as declared with 'make_file' at external environment configuration. To ensure these shared actions are imported in the final make file part of the package, place their declarations within text tokens - ";WF_START", ";WF_END"

Passing Variables

You may have an action which takes an argument that is, perhaps, return of another action. Than you can declare it as variable with 'result' index and access it via '@' as following:

action[get_make][name]=get_make_file
action[get_make][order]=0
action[get_make][tag]=config
action[get_make][result]=make_file
action[config_depend][name]=configure_all_dependencies
action[config_depend][order]=1
action[config_depend][tag]=config
action[config_depend][param][]=@make_file

Here, the action 'get_make' retrieves the make file and than store it in the variable "make_file" that is later reused as an argument in the action - "config_depend"

Note: sharing variables are only supported for command - execute-action

Deployment Cycles

DEV (development)

When deploying artifacts in DEV mode, it will link the artifacts to its appropriate repositories in the working directory, so changes you make are updated in the working directory. By convention, the location of working version of the artifact is located at 'work_dir'/'artifact type'/'artifact name'. This can be overwritten by 'work_dir', 'work_dir_name' attributes of artifact in blueprint or artifact external configurations

TEST

When deploying artifacts in TEST mode, it will copy the artifacts from the appropriate package located in your local distribution directory, so before delivering to client the package is tested. The distribution dir is configured in the 'dist' variable in external tool configurations. By convention, the packages are stored in 'dist'/'artifact type'/'package name'

STAGE

When deploying artifacts in STAGE mode, it will clone the artifacts from external git hub repository per blueprint(i.e make file) provided. This may be alternative method to deliver your artifacts. Instead of sending the package, you can only provide the blueprint(i.e. make file) and your client can deploy it in STAGE mode that way retrieving all of your artifacts from git hub repository.

Command Arguments API

All command arguments are parsed and processed into following variables available to retrieve any time via drush_get_context('VAR_NAME');

  • WF_ASSET_NAME: the name of artifact taken from first arg. This can also be set with tag --artifact
  • WF_ASSET_TYPE: the type of artifact taken form second arg
  • WF_PROFILE_NAME: the name of the profile make file
  • WF_ASSET_VER: the version of package taken from flag '--ver'
  • WF_ASSET_MIN: whether the dependencies are included taken from flag '--min'
  • WF_ASSET_BRANCH: the branch of the asset taken from flag '--branch'
  • WF_ASSET_ENV: the deployment mode for asset such as dev, stage, test,etc
  • WF_ASSET_MAKE: make file of asset provided by user via flag '--make'.
  • WF_ASSET_PACKAGE: the path to the package directory or compressed package of asset specified by flag --package. It defaults to DIST/"package dir" or "current dir"/package
  • WF_ALIAS: the alias record all loaded by name specified by user input with '@'
  • WF_INPUT_PARAM: list of arguments to pass into action if called dynamically from execute-action DRUSH command
  • WF_INPUT_TAG: list of tags assigned to actions to be executed via DRUSH command - configure-asset
  • WF_INPUT_ACTION:list of action to execute via DRUSH command - configure-asset

Variable API

Here are list of available env variables that are initiated at the bootstrap time read from external configurations

  • WF_ROOT: This is location of WorkFlow root. It is used for cache as well as default location for blueprints, distributions, etc if not specified other location. The default dir for root of WorkFlow tool is $HOME/workflow
  • WF_WORK_DIR: This is location of all artifact repositories where the development takes place and is specified in the environment configurations with 'work_dir'. If not specified, then defaults to $HOME/workflow/work-dir
  • WF_MAKE_ROOT: This is location of all artifact blueprints reside and is specified in the environment configurations with 'make_root'. If not specified, then defaults to $HOME/workflow/blueprints
  • WF_CACHED_MAKE: the make file gets cached in this location
  • WF_DIST: This is location where all the built packages are kept. It is specified in the environment configurations with 'dist'. If not specified, then defaults to $HOME/workflow/dist
  • WF_DEFAULT_MAKE: This is blueprint(make file) for site installs as well as any shared actions, artifact info. It is specified in the environment configurations with 'make_file'. If not specified, then defaults to make_root/build-drupal-core.make
  • WF_DB_USER: This database username used for site installs. It is specified in the environment configurations with 'db_user'. If not specified, then defaults it defaults to "root"
  • WF_DB_PWD: This database user password for site installations. It is specified in the environment configurations with 'db_pwd'. If not specified, then defaults to "password"
  • WF_PACKAGE_PREFIX: The prefix added to the package name of asset configurable at the external environment configuration at build process
  • WF_ARTIFACTS:List of artifacts as declared in the artifacts.drushrc.php that is merged with artifact declarations in the make file before storing into this variable
  • WF_ARTIFACTS_TYPES: Array of different artifact types - widget, theme, etc that can be specified in external environment configurations via '$artifact_types'. If not specified, then defaults to two types - "widget","theme"

Tags

Tags Overview

WorkFlow has actions that are tasks to do something such as configure audio, enable editor, etc. A larger task such as configure video may take many smaller tasks(i.e download lib, enable modules, link library,etc), so to better handle this we have tags. You can assign one or many tags to each action in that way grouping common tasks together that can be triggered based on one tag. In another words, you can DRush command 'execute-action'[ea] and provide a tag which will take and execute all actions associated with this tag.

Actions are also declarative and are declared in blueprint(make file). This provides a way for you repeat actions whenever you wish. It also provides way for you to share actions. In WorkFlow, the shared actions are imported at the build process into all packages, so you can reuse them. Here is how you can call them

drush ca @ALIAS --tag=TAG-NAME --asset=ASSET-NAME
drush ca @ALIAS --tag=TAG-NAME --make=path/to/make/file
drush ca @ALIAS --tag=TAG-NAME --package=path/to/package

So, by providing tag, you would call different tasks

How to add new Artifact(i.e. widget,theme,etc)?

In the following are steps on configuring new widget(i.e. map widget)

  1. Create repository in Github.com for the new artifact
  2. Declare artifact in the artifact configuration file located at $HOME/.drush/artifact.drushrc.php or $HOME/workflow/artifact.drushrc.php. Please, see details in Artifacts section
  3. From here, you can do one of the two:

    1. Do it manually:

      1. Create working directory for the new artifact
      2. Add info and other files necessary for Drupal module or theme
      3. Hook the new working directory with github. In the new working directory:
        1. initalize git repo(git init)
        2. add new files(git add .)
        3. commit (git commit -m "first commit")
        4. add remote repository(git remote add origin url-of-git-repo.git)
        5. push it(git push origin master)

    2. Call action to create boilerplate and hook the git repo automatically for you


      drush ea create_artifact --param=ARTIFACT-NAME

How to spin new Drupal instance?

  1. Configure Virtual Host

    Virtual host needs to be configured to service the new Drupal Instance. On how to configuring Virtual Host, it is out of scope for this documentation. Please, see post Installing Apache On Mac in section on "Virtual Host" or refer any other resources on how to configure virtual host

  2. Configure Database

    You have to create new Database name, perhaps DB Username and Password as well.

    1. Login in your mysql
      mysql -u root -p
    2. Create Database
      create database DB-NAME;
    3. Create User and Password grant all privileges on DB-NAME.* to USER-NaME@'localhost' identified by 'USER-PASS';
    4. Flush It flush privileges;

  3. Configure Alias Settings

    Besides default Drush $alias settings, here the addition needed:

    • Db User(db-user) 'db-user' => 'dev_metronic',
    • Db Password(db-pwd) 'db-pwd' => 'pass',
    • Db Name(db-name) 'db-name' => 'dev_metronic_db',
    • [OPTIONAL]Default Make File(make-file) 'make-file' => 'path/to/default/build-drupal-core.make',
    For more info, please, see section on Alias Configurations

  4. Configure Local Lookup Table

    To ensure the request to the new instance are routed correctly, update your local look up table(ie. for mac /etc/hosts) 127.0.0.1 URI

  5. Spin It
    drush is @ALIAS-NAME

How to Build Asset For First Time

  1. Create Asset Blueprint

    Every asset has its blueprint the make file. So, start by creating make file for your new asset about to build. If you are not going to provide this make file from command line with '--make' option, then WorkFlow will look for one at $BLUEPRINT/type(s)/PRE-FEX-TYPE-NAME-VER.make (see more in Asset Naming section)

    In the make file, declare the artifacts you like to be included. In the following is make file for asset - Drupal Metronic Theme that will help you start

    ...
    ;;WIDGETS
    widget[lib][name]=lib
    widget[lib][order]=1
    widget[lib][ver]=v.0.2.2
    widget[sample_data][name]=sample_data
    widget[sample_data][order]=2
    widget[sample_data][ver]=v.0.1
    widget[metronic_theme][name]=metronic_theme
    widget[metronic_theme][order]=4
    widget[metronic_theme][ver]=v0.0.0
    widget[utils_ckeditor][name]=utils_ckeditor
    widget[utils_ckeditor][order]=5
    widget[utils_ckeditor][ver]=v0.1.0

    ;;THEMES
    theme[bootstrap][name]=bootstrap
    theme[bootstrap][order]=1
    theme[bootstrap][ver]=v0.1
    theme[metronic][name]=metronic
    theme[metronic][order]=2
    theme[metronic][ver]=v0.0.0
    ...
    ;;ACTIONS
    ...
    projects[jquery_update][version] = 2.x-dev
    projects[] = context
    projects[] = ctools
    projects[] = features
    projects[] = strongarm
    projects[] = features_extra
    projects[] = views projects[] = entity projects[] = pathauto projects[uuid][version] = 1.x-dev projects[uuid_features][version] = 1.x-dev

    Here we add some shared dependencies such as lib, sample_data for exporting via features, utils_ckeditor for our custom ckeditor with 20 most popular free CKEditor plugins. The metronic_theme is custom module providing functionality to drupal theme - Metronic an asset we are building. As you see version is v0.0.0 is assigned because we utilized action to build the boilerplate that adds the v0.0.0 at the generation.

    The asset - Drupal Theme Metronic is going to have two themes - the base theme Bootstrap and custom Metronic. Actions will go in this make file later once we ready to deploy. At last, all other dependencies and their versions are included

  2. Build Asset

    Once you have the make file, you ready to build

    drush ba metronic theme --ver=0.1

    --OR--

    drush ba metronic theme --make=themes/prefix-type-metronic-0.1.make --ver=0.1

    Here we are building asset of type theme named metronic. For the first one, WorkFlow will try to locate the make file per conventions with version 0.1 attached at the end. Please, see sections on Asset Naming and Conventions. For the second, you can specify relative path to make file from your $MAKE_ROOT as declared in the environment configurations

    3. On success, look into $DIST/TYPE(s)/ directory for the newly built asset

How to deploy asset?

Once you have the package built, spin up Drupal instance and lets deploy. In the following, there are three different ways to deploy based on the method you provide make file

  1. Directly Providing Blueprint via "--make"
    drush da @dev-metronic --make=themes/PREFIX-theme-metronic-0.1.make --env=dev
  2. By Specifying Package via "--package"

    since each package contains make file, we can specify the package as following

    drush da @dev-metronic --package=themes/designssquare-com-theme-metronic/designssquare-com-theme-metronic-0.1 --env=test

    --OR--

    drush da @dev-metronic --package=themes/designssquare-com-theme-metronic/designssquare-com-theme-metronic-0.1.tar.gz --env=test

    Here again, we specify path to package direcotry or compressed package relative to distribution directory as specified in WorkFlow evironment variable "dist"

  3. By Specifying Asset via "--asset" & "--ver" & "--branch"
    drush da @dev-metronic --asset=metronic --ver=0.1 --env=stage

    this will use the artifact configurations(see Artifacts Configurations) with conventions(see Conventions) to locate make file for the particular asset.

    4. From the above, you can see the "env" was provided as well. Please, see section "Deployments" about three different ways to deploy based on "env"

Troubleshooting

TIP: Turn on logging

When troubleshooting issue, its helpful to turn on logging. There are two levels important to mention

  • notice: To turn on add flag '--verbose' to the Drush command. This will print all messages of progress as the execution takes place
  • debug: To turn on add flag '--debug' to the Drush command. Besides 'notice' message, this will print all messages including ones associated with Drush execution
